3. Project Overview

FastTrack is a full-stack web application combining calorie tracking with an intelligent intermittent fasting timer. The app adapts to real user behavior - if a user eats outside their scheduled window, the fasting timer automatically recalculates from that meal's timestamp. An AI assistant powered by OpenAI provides personalized coaching, daily summaries, and nutritional guidance based on the user's actual logged data.

Primary Goal

Portfolio/interview demo piece demonstrating proficiency across a professional tech stack: React, Tailwind CSS, Node/Express, PostgreSQL, Firebase Auth, OpenAI, and external API integration (USDA FoodData Central).

7. Database Schema

Three tables in the fasttrack database on local PostgreSQL 18. All queries use parameterized $1, $2 syntax via the pg library.

CREATE TABLE user_profiles (
  id                  SERIAL PRIMARY KEY,
  firebase_uid        VARCHAR(128) UNIQUE NOT NULL,
  email               VARCHAR(255),
  display_name        VARCHAR(100),
  daily_calorie_goal  INTEGER DEFAULT 2000,
  fasting_protocol    VARCHAR(20) DEFAULT '16:8',
  fasting_hours       INTEGER DEFAULT 16,
  eating_hours        INTEGER DEFAULT 8,
  timezone            VARCHAR(50) DEFAULT 'America/New_York',
  ai_summary_date     DATE,
  ai_summary_text     TEXT,
  created_at          TIMESTAMP DEFAULT NOW()
);

CREATE TABLE meal_logs (
  id           SERIAL PRIMARY KEY,
  user_id      INTEGER REFERENCES user_profiles(id) ON DELETE CASCADE,
  food_name    VARCHAR(255) NOT NULL,
  calories     INTEGER NOT NULL,
  protein_g    DECIMAL(6,1),
  carbs_g      DECIMAL(6,1),
  fat_g        DECIMAL(6,1),
  quantity     DECIMAL(6,2) DEFAULT 1,
  unit         VARCHAR(50),
  eaten_at     TIMESTAMP DEFAULT NOW(),
  created_at   TIMESTAMP DEFAULT NOW()
);

CREATE TABLE fasting_sessions (
  id             SERIAL PRIMARY KEY,
  user_id        INTEGER REFERENCES user_profiles(id) ON DELETE CASCADE,
  fast_start     TIMESTAMP NOT NULL,
  fast_end       TIMESTAMP,
  target_hours   INTEGER NOT NULL,
  completed      BOOLEAN DEFAULT FALSE,
  broken_early   BOOLEAN DEFAULT FALSE,
  created_at     TIMESTAMP DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_meal_logs_user_id ON meal_logs(user_id);
CREATE INDEX idx_meal_logs_eaten_at ON meal_logs(eaten_at);
CREATE INDEX idx_fasting_sessions_user_id ON fasting_sessions(user_id);
CREATE INDEX idx_fasting_sessions_fast_end ON fasting_sessions(fast_end);

9. Key Implementation Details

9.1 Adaptive Fasting Timer (mealsController.js)

When POST /api/meals is called:

  1. Insert meal into meal_logs with current timestamp
  2. Find active fasting session (fast_end IS NULL)
  3. If active session exists and fast_start is within 30 minutes (grace window): just shift fast_start forward to current time
  4. If active session exists and older than 30 min: close it (set fast_end, calculate completed/broken_early), open new session from current timestamp
  5. If no active session: create new one from current timestamp
  6. Return meal + fasting session + timeRemainingSeconds in response

All of this runs inside a database transaction for atomicity.

9.2 USDA FoodData Central (foodController.js)

Endpoint: https://api.nal.usda.gov/fdc/v1/foods/search

Nutrient IDs: 1008 (calories), 1003 (protein), 1005 (carbs), 1004 (fat)

Filters: SR Legacy, Survey (FNDDS), Foundation data types. Excludes 0-calorie items. Cleans up ALL CAPS descriptions to Title Case.

9.3 AI Chat (aiController.js)

System prompt includes user's live data: name, calorie goal, fasting protocol, today's calories/macros, fasting status, time remaining, streak, recent meals. Uses SSE streaming (text/event-stream) with gpt-4o-mini. Chat history limited to last 10 messages.

9.4 Firebase Auth Flow

Frontend: Firebase SDK handles login, returns JWT. Token attached as Authorization: Bearer <token> on every API call via api.js.

Backend: authMiddleware.js calls admin.auth().verifyIdToken(), sets req.user with decoded UID. All controllers look up user_profiles by firebase_uid.

12. Known Issues & Gotchas

  • curl on Windows: Fails to reach USDA API due to TLS certificate revocation check (CRYPT_E_NO_REVOCATION_CHECK). This is a Windows curl issue only - Node fetch works fine. The app is not affected.
  • Vite port: If port 5174 is occupied, Vite picks the next available port. If this happens, update CLIENT_URL in server/.env and restart the backend so CORS allows the new origin.
  • Firebase token expiry: Tokens expire after 1 hour. The client SDK auto-refreshes them via getIdToken() on each API call.
  • Timezone handling: Stored in user_profiles but not actively used in queries yet - all timestamps are UTC.
